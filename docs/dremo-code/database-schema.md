# Proposed Dremo Database Schema

Status: proposed. This file does not create SQL migrations.

The future Dremo schema should use dedicated `dremo_*` tables instead of stretching the legacy `execution_sessions` and `execution_logs` tables. Existing execution tables can remain as compatibility data during migration.

## Policy Direction

| Actor | Select | Insert | Update |
| --- | --- | --- | --- |
| User frontend | Own tasks/events/artifacts only. | No trusted runtime inserts. | Limited action requests through API only, not direct table update. |
| Dremo backend / service role | All Dremo rows needed for execution. | Yes. | Yes. |
| Admin tools | Explicit audited admin access only. | Case-by-case. | Case-by-case. |

## dremo_tasks

Purpose: canonical task record and lifecycle state.

Important columns:

| Column | Purpose |
| --- | --- |
| `id` | Primary task id. |
| `user_id` | Owner from Supabase auth. |
| `agent_slug` | Current entry point, initially likely `code-architect`. |
| `title` | User-facing task title. |
| `prompt_snapshot` | Sanitized snapshot of original user request. |
| `workspace_source` | JSON source descriptor for repo/upload/template. |
| `status` | Task state machine value. |
| `billing_state` | Trusted billing state. |
| `created_at` / `updated_at` | Audit timestamps. |
| `completed_at` | Completion timestamp. |

RLS direction:

| Role | Policy |
| --- | --- |
| `authenticated` | Select rows where `auth.uid() = user_id`. |
| `authenticated` | No insert/update/delete. |
| `service_role` | Insert/update/select. |

Relationship to legacy tables: `execution_sessions` can map to a simplified view of task id, status, agent, cost, and timestamps during transition.

## dremo_task_events

Purpose: ordered server-owned structured event stream.

Important columns:

| Column | Purpose |
| --- | --- |
| `id` | Primary event id. |
| `task_id` | Parent task. |
| `user_id` | Owner duplicate for fast RLS and queries. |
| `sequence` | Monotonic per task. |
| `type` | Event type. |
| `channel` | Rendering channel. |
| `severity` | Debug/info/warning/error. |
| `payload` | JSON payload. |
| `redaction_state` | `clean`, `redacted`, `quarantined`. |
| `created_at` | Server timestamp. |

RLS direction:

| Role | Policy |
| --- | --- |
| `authenticated` | Select own events only. |
| `authenticated` | No insert/update/delete. |
| `service_role` | Insert/select/update for redaction/quarantine only. |

Relationship to legacy tables: replaces `execution_logs` for Dremo. Legacy text logs may be generated from Dremo events for compatibility if needed.

## dremo_approvals

Purpose: user approval requests and decisions for risky actions.

Important columns:

| Column | Purpose |
| --- | --- |
| `id` | Approval id. |
| `task_id` | Parent task. |
| `user_id` | Owner. |
| `action_type` | Command, network, file write, external publish, billing escalation. |
| `presented_text` | Exact text shown to user. |
| `risk_level` | Low/medium/high/critical. |
| `status` | Pending/approved/rejected/expired. |
| `resolved_at` | Decision timestamp. |

RLS direction: users can read own approvals, but decisions should be submitted through authenticated API so the backend can validate scope and append events.

## dremo_artifacts

Purpose: metadata for generated patches, reports, logs, screenshots, and downloads.

Important columns:

| Column | Purpose |
| --- | --- |
| `id` | Artifact id. |
| `task_id` | Parent task. |
| `user_id` | Owner. |
| `kind` | Patch, report, log, screenshot, archive, PR metadata. |
| `storage_key` | Private object storage path. |
| `name` | User-facing name. |
| `size_bytes` | Size guardrail. |
| `content_hash` | Integrity check. |
| `created_at` | Timestamp. |

RLS direction: users can select own artifact metadata. Downloads should use short-lived signed URLs generated by API.

## dremo_task_credit_ledger

Purpose: Dremo-specific mapping to existing credit ledger entries, not a competing wallet.

Important columns:

| Column | Purpose |
| --- | --- |
| `id` | Mapping id. |
| `task_id` | Parent task. |
| `user_id` | Owner. |
| `credit_ledger_entry_id` | Existing `credit_ledger` entry. |
| `entry_kind` | Reserve, charge, release, refund, adjustment. |
| `amount_credits` | Signed amount. |
| `idempotency_key` | Duplicate prevention. |

RLS direction: users can read own mappings if useful. Writes are service-role only.

Relationship to existing ledger: this should reference the canonical Lumixia credit ledger rather than create a second wallet.

## dremo_sandbox_sessions

Purpose: track sandbox lifecycle and isolation metadata.

Important columns:

| Column | Purpose |
| --- | --- |
| `id` | Sandbox session id. |
| `task_id` | Parent task. |
| `provider` | Docker, E2B, Daytona, Fly Machines, Cloud Run, Kubernetes, or future provider. |
| `status` | Creating, ready, running, cleaning, destroyed, failed. |
| `resource_limits` | CPU, memory, disk, timeout. |
| `network_policy` | Egress policy snapshot. |
| `created_at` / `destroyed_at` | Lifecycle timestamps. |

RLS direction: users may read sanitized session status. Provider internals and credentials remain backend-only.

## dremo_model_runs

Purpose: model call metadata for cost, debugging, and quality control.

Important columns:

| Column | Purpose |
| --- | --- |
| `id` | Model run id. |
| `task_id` | Parent task. |
| `user_id` | Owner. |
| `provider` | Model provider. |
| `model` | Model id. |
| `purpose` | Planning, coding, review, repair, summarization. |
| `input_tokens` / `output_tokens` | Usage metrics. |
| `cost_minor` | Internal cost estimate if tracked. |
| `created_at` | Timestamp. |

RLS direction: users may see summarized model usage, but raw prompts/responses should be redacted or hidden when they include private code or secrets.

## Suggested Indexes

| Table | Index |
| --- | --- |
| `dremo_tasks` | `(user_id, created_at desc)` |
| `dremo_task_events` | `(task_id, sequence)` unique |
| `dremo_task_events` | `(user_id, created_at desc)` |
| `dremo_approvals` | `(task_id, status)` |
| `dremo_artifacts` | `(task_id, created_at desc)` |
| `dremo_task_credit_ledger` | `(task_id, idempotency_key)` unique |
| `dremo_sandbox_sessions` | `(task_id, created_at desc)` |
| `dremo_model_runs` | `(task_id, created_at desc)` |

## Migration Strategy From Legacy Execution Tables

| Step | Direction |
| --- | --- |
| 1 | Keep `execution_sessions` and `execution_logs` hardened and read-only to users. |
| 2 | Add `dremo_*` tables with service-owned writes. |
| 3 | Create Dremo API stub that writes `dremo_tasks` and `dremo_task_events`. |
| 4 | Update workspace UI to read Dremo events. |
| 5 | Optionally create compatibility views for old execution surfaces. |
| 6 | Rename Code Architect AI only after Dremo runtime, credits, and event stream are live. |
